# Internationalization Strategy

## Scope

Проект использует две локали:

- `ru` - default locale и основная редакционная база
- `en` - полная англоязычная версия публичного сайта и SEO-контур

## Product Rules

- Все публичные страницы должны иметь устойчивые локализованные URLs вида `/{locale}/...`
- Корневой путь `/` редиректит на `/ru`
- Переключатель языка должен сохранять текущий путь и менять только locale prefix
- Админка остается технически single-entry, но контент внутри нее редактируется в двух языках
- Метаданные, `sitemap` и `canonical` должны быть locale-aware

## Public Layer

- Текст публичных страниц не должен быть собран из разрозненных inline-строк
- Локализация публичного UI и page copy живет в отдельном content layer
- URL и контент должны совпадать по локали, без скрытого смешения языков
- Gallery, artwork page, about, exhibitions и contacts обязаны рендериться с учетом locale

## Admin Layer

- Редактор вводит данные отдельно для `ru` и `en`, если поле предназначено для контентного текста
- Слои формы и сервиса должны явно различать multilingual fields и shared fields
- Общие технические поля остаются single-source of truth: `slug`, `status`, `sortOrder`, `isPublished`, media metadata
- Не дублировать логические сущности только ради языков

## Data Model

Рекомендуемый принцип хранения:

- Shared data - одна запись на сущность
- Localized content - отдельные поля `*Ru` / `*En`
- Relations и operational fields - без локализации

Примеры:

- `Category.titleRu`, `Category.titleEn`
- `Category.descriptionRu`, `Category.descriptionEn`
- `Artwork.titleRu`, `Artwork.titleEn`
- `Artwork.descriptionRu`, `Artwork.descriptionEn`
- `Artwork.seoTitleRu`, `Artwork.seoTitleEn`
- `Artwork.seoDescriptionRu`, `Artwork.seoDescriptionEn`
- `Exhibition.titleRu`, `Exhibition.titleEn`
- `Exhibition.descriptionRu`, `Exhibition.descriptionEn`
- `SiteSettings.heroTitleRu`, `SiteSettings.heroTitleEn`
- `SiteSettings.artistStatementRu`, `SiteSettings.artistStatementEn`

## Content Rules

- `slug` всегда единый и не зависит от локали
- Английская версия не должна быть машинным зеркалом русского текста, если смысл требует адаптации
- Названия серий, выставок и картин должны быть семантически эквивалентны, а не дословно равны любой ценой
- Для SEO приоритет у читаемости и точности, а не у буквального перевода
- Если перевод временно отсутствует, fallback допускается только в draft/transition stage
- На публичном сайте не смешивать русский и английский в одном блоке

## Migration Notes

### Phase 1

- Ввести locale routing и `en/ru` public shells
- Зафиксировать shared i18n helpers и path builders
- Оставить текущие mock/content fallbacks как transitional source

### Phase 2

- Перенести основные сущности в multilingual schema
- Обновить admin forms на `ru/en` поля
- Перевести public loaders на чтение из БД по locale

### Phase 3

- Удалить transitional copy layer там, где данные уже живут в БД
- Проверить `sitemap`, `canonical`, metadata и internal links
- Провести контентную ревизию на mixed-language fragments

## Guardrails

- Не добавлять третью локаль без отдельного решения
- Не смешивать UI language и content language в одном контракте без явного naming
- Не хранить переводы в JSON-строках внутри одного технического поля, если поле редактируется через админку
- Не ломать опубликованные slugs при локализации

## Acceptance Criteria

- Пользователь открывает `/ru` и `/en` как полноценные версии сайта
- Админка позволяет редактировать двуязычный контент без ручного вмешательства в БД
- SEO-метаданные и sitemap корректно отражают обе локали
- В production нет страниц, где перевод зависит только от hardcoded UI copy
